# AI-Assisted Counseling Dialogue and Case Conceptualization System

This repository contains a classroom/demo prototype for an AI-assisted
counseling documentation workflow.

The system is designed for counselors, not clients. A counselor enters
client-provided content, reviews structured micro-summaries and crisis-language
metadata, and may generate a draft case conceptualization report for further
professional review.

The central boundary is intentionally narrow: AI assists documentation and
conceptualization. It does not diagnose, prescribe, replace therapy, replace
medical care, or make final clinical decisions.

## Project Status

The current implementation supports an end-to-end demo flow:

- Case and durable session creation
- Counselor-operated conversation workspace
- Browser-based speech recognition input assistance
- AI-assisted response generation
- Per-turn structured summaries
- Conservative crisis-language metadata
- History view with manual session titles and archive/unarchive actions
- Report review aids for emotion, themes, summaries, and crisis occurrence
- Manual report-data input
- AI draft generation for case conceptualization
- Read-only report preview
- Browser print / save-as-PDF friendly report view

This is not a production clinical system. Real clinical deployment would require
privacy, consent, governance, security, evaluation, and operational work that is
outside the current prototype.

## Safety Boundaries

All report, agent, API, and UI behavior should preserve these rules:

- The counselor is the final reviewer and decision-maker.
- The client is not a direct user of the system.
- AI output is draft support material only.
- The system must not provide psychological or psychiatric diagnosis.
- The system must not provide medication, dosage, stopping-medication,
  prescription, or medical advice.
- The system must not generate formal treatment instructions, formal risk
  assessments, or formal safety plans.
- Crisis metadata is a review prompt for the counselor, not a clinical risk
  determination.
- Demo, automated tests, and manual provider smoke tests should use synthetic or
  fully de-identified data.

Report-related views keep a fixed AI-draft disclaimer supplied by application
code. The disclaimer is not generated by an LLM.

## Architecture

```text
frontend/                  React + Vite counselor-facing UI
backend/                   FastAPI backend
backend/agents/            Conversation, summary, crisis, and report agents
backend/routers/           HTTP API routers mounted under /api
backend/database/db.py     SQLite + aiosqlite persistence layer
backend/models/            Report draft schema models
docs/                      Status, contracts, safety notes, and runbooks
```

### Backend

The backend is a FastAPI app with SQLite persistence. It stores cases, sessions,
messages, summaries, and report drafts. Session metadata is intentionally safe
and operational: titles and archive status are counselor-controlled metadata and
must not be derived from clinical content.

The implemented agent modules are:

- `crisis_agent.py`: conservative crisis-language detection with fail-safe
  fallback behavior.
- `summary_agent.py`: per-turn structured summaries.
- `conversation_agent.py`: AI-assisted counselor-facing conversation support.
- `analysis_agent.py`: report generation, deterministic Report Schema v2 draft
  generation, provider-mode parsing, and safety validation.

### Frontend

The frontend is a React + Vite app with pages for:

- Conversation workspace
- Report workspace
- History and session management
- Settings and system guidance

Frontend tests use Vitest, React Testing Library, and jsdom with mocked API
helpers. They do not call the live backend, live providers, or the network.

## Technology Stack

- Frontend: React, Vite, React Router, Tailwind CSS, Recharts, Vitest
- Backend: Python, FastAPI, Pydantic, SQLite, aiosqlite, pytest
- LLM providers: Gemini and Groq, configured through backend environment
  variables
- Default report draft mode: deterministic and provider-free unless explicitly
  configured otherwise

## Getting Started

Run the backend and frontend in separate terminals.

### Backend (Windows User)

```powershell
1.	cd backend
2.	python -m venv .venv
3.	.\.venv\Scripts\Activate.ps1
4.	python -m pip install --upgrade pip
5.	pip install -r requirements.txt
6.	copy .env.example .env
7.	# Please fill in the required settings in the .env file first, e.g., `GEMINI_API_KEY`, `DATABASE_PATH`, etc., before continuing to start the backend.
8.	python -m uvicorn main:app --reload
```

### Backend (Linux / Mac User)

```powershell
1.	cd backend
2.	python3 -m venv .venv
3.	source .venv/bin/activate
4.	python -m pip install --upgrade pip
5.	pip install -r requirements.txt
6.	cp .env.example .env
7.	# Please fill in the required settings in the .env file first, e.g., `GEMINI_API_KEY`, `DATABASE_PATH`, etc., before continuing to start the backend.
8.	python -m uvicorn main:app --reload
```

The backend runs at:

```text
http://127.0.0.1:8000
```

Health check:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/health
```

### Frontend

```powershell
npm run dev
```

## Configuration

Backend configuration lives in `backend/.env.example`. Copy it to
`backend/.env` for local development and fill in only the settings you need.

Common settings include:

- `GROQ_API_KEY`: used by Groq-backed crisis/summary paths and optional report
  provider mode.
- `GEMINI_API_KEY`: used by Gemini-backed conversation/report paths.
- `DATABASE_PATH`: SQLite database path.
- `CONVERSATION_MAX_TOKENS`: conversation response token budget.
- `MIN_TURNS_FOR_REPORT`: minimum turn count for report generation behavior.
- `REPORT_V2_PROVIDER_MODE`: `deterministic` or `provider`.
- `REPORT_V2_PROVIDER`: `gemini` or `groq` when provider mode is enabled.
- `REPORT_V2_API_KEY`: optional report-specific provider key override.
- `REPORT_V2_FALLBACK_ENABLED`: optional report-provider fallback, disabled by
  default.

Do not commit `.env`, API keys, provider responses, prompts, local database
files, or clinical data.

## Testing

Run deterministic backend tests from the repository root:

```powershell
python -m pytest backend\tests -q --basetemp=.tmp_pytest_backend -p no:cacheprovider
```

Run frontend tests and build checks from `frontend/`:

```powershell
cd frontend
npm run test
npm run build
```

Automated tests should remain deterministic and network-free. Live provider
checks belong in manual smoke-test workflows only and should use synthetic or
fully de-identified data.

## Privacy and Storage Notes

The frontend must not store clinical message content, summaries, report text,
report drafts, manual input, AI-generated JSON, crisis reasons, titles, or
session metadata in browser storage.

Current browser storage usage is intentionally narrow:

- `localStorage`: theme preference only (`ai-psych-theme`)
- `sessionStorage`: active case/session identifiers only

Report draft content and manual input are handled through backend APIs and
React state, not browser persistence.

## Current Limitations

- No production authentication or role-based access control.
- No client-facing workflow.
- No counselor final-report editing/review status workflow yet.
- No true server-side PDF export, the current report view supports browser
  print / save-as-PDF only.
- No hard-delete session workflow; data-retention policy must be designed first.
- No production deployment, security hardening, or clinical validation.
